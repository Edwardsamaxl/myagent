# Agent 全链路设计评审门禁

> 角色：PM 维护。目标：任何**契约**或**对外行为**变更不得「先代码、后文档」；与 [coordination.md](../coordination.md) §3 硬门禁对齐并略作 Agent 场景补充。

---

## 1. 与 §3.12 的关系（唯一真理源）

仓库已规定 **契约变更放行规则**，全文见 [coordination.md §3.12](../coordination.md)：

- **触发范围**：错误码集合、HTTP 状态语义、接口字段名/结构（请求或响应）变化。  
- **放行顺序**：**先**更新 `coordination.md` §3 **对应小节**，**再**合并实现代码；未更新文档视为未对齐，**不放行**。  
- **执行责任**：PM 把关；D 负责 API 与错误码落地；A/B/C 改动前先引用 §3 确认口径。

本文 **不重复改写** §3.12，仅将「Agent 全链路设计」纳入同一套流程。

---

## 2. Agent 场景下的扩展触发范围（设计稿也需过门禁）

下列内容一旦要**落进实现**或**改变用户/脚本可见行为**，必须先反映到 `coordination.md` §3（新增小节或扩写现有小节），再写代码：

| 变更类型 | 应写入 §3 的位置（示例） |
|----------|---------------------------|
| `/api/chat` 请求/响应字段增删改 | §3.8；若与 RAG 子结构共享，交叉引用 §3.9 |
| 工具调用对外暴露（如新 REST 或新顶层字段） | §3 新增子节或 §3.8 扩展，并与错误码表 §3.6 对齐 |
| `retrieval_hits` 主字段名/必选结构变化 | §3.9（当前要求 5 字段名稳定） |
| 生成侧输出字段、拒答 reason 集合变化 | §3.14 |
| 评估/trace 对外字段约定变化 | §3.11 或新增 §3.x（由 PM 与 E 商定小节号） |

**纯内部重构**（私有函数拆分、无 JSON 变化、无新错误码）可不经 §3 扩写，但须在 PR 说明中写清「无契约变更」。

## 3. 设计评审检查清单（合并 PR 前）

责任人：PR 作者自检 + PM 抽检。

1. [ ] 是否修改了 HTTP/API、错误码、trace 或评估 JSON 的**对外形状**？  
   - 是 → [coordination.md](../coordination.md) §3 是否已有**同级**文字描述？  
2. [ ] Agent 循环、RAG 注入、工具返回是否引入**新的用户可见 reason / code**？  
   - 是 → §3.6 / §3.14（或约定小节）是否已更新？  
3. [ ] 是否与 [mvp-scope.md](mvp-scope.md) 的「明确不做」冲突？  
   - 若冲突，是否已升格为周计划/协调说明中的正式范围变更？  
4. [ ] 若多角色并行曾分歧，[merge-decisions.md](merge-decisions.md) 是否已更新裁决结果？

## 4. 例外与升级

- **紧急热修**（线上安全/可用性）：允许先最小修复，但 **24h 内** 补写 §3 与 PR 说明，否则 PM 在 `coordination.md` §2「与计划差异」或评审记录中记一笔技术债。  
- **文档与代码谁为准**：以 [coordination.md](../coordination.md) 开篇约定为准——与周计划字面冲突时，以本文档「与计划差异」及代码事实为准，并应回写。
